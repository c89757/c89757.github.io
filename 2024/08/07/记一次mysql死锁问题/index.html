记一次mysql死锁问题

2024-08-07 MySQL MySQL, questions 0 评论

背景

线上生产报障，查看程序日志，发现某条更新sql报错，报错信息dead lock

排查

在Mysql客户端执行命令：show engine innodb status，可以查看到死锁信息（默认只会记录最后一次发生死锁的信息）

复现

  • 事务隔离级别

REPEATABLE-READ

  • 新建测试表
  • 插入测试数据
  • 复现
时刻 事务1 事务2
T1 begin; begin;
T2 SELECT * FROM test_deadlock force index(idx_tenant_id) where tenant_id = ‘123’ and account_id in (‘X’) for update
T3 SELECT * FROM test_deadlock force index(idx_message_id) where message_id = 200 and account_id in (‘Z’,’X’) for update
T4 阻塞
T5 SELECT * FROM test_deadlock force index(idx_tenant_id) where tenant_id = ‘123’ and account_id in (‘Z’) for update

show engine innodb status执行后的日志:



根本原因就是并发情况下，索引顺序不一致，导致加锁顺序不一致产生的死锁




By the way! 即使事务2的语句改为只查询’Z’也会发生死锁：

SELECT * FROM test_deadlock force index(idx_message_id) where message_id = 200 and account_id in (‘Z’) for update；

这是因为：二级索引上包含的信息，并不能判断条件是否成立，所以需要获取下一行数据，继续回表扫描。

解决方案

  • 诱导mysql执行器走同一个索引
  • 从业务上面，避免这种场景
